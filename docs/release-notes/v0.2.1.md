# secdat v0.2.1 Release Notes

## Highlights

- Adds the subcommand registry-backed help surface and keeps shell completion aligned with the CLI command table.
- Expands store v2 support around domain-entry/secret-object links, `@UUID` source references, object status inspection, refcount repair, orphan/dangling cleanup, and migration finalization guidance.
- Clarifies sandbox injection attributes: `--sandbox-injectable` is a bulk policy gate, while plain pattern filters remain direct visible-key selectors.
- Improves Japanese help coverage and regression checks for user-facing help text.

## C SDK and CLI

- The C project version is `0.2.1`.
- `secdat id KEYREF` prints the resolved v2 secret object UUID without reading the value.
- `secdat secret status UUID` reports non-secret v2 object metadata from the current domain/store object view.
- `store migrate`, `fsck --format v2`, `store finalize-migration`, and `gc --format v2` now have clearer migration and cleanup guidance in help and docs.

## Bindings

- Python, Rust, and Node binding package versions now track `0.2.1`.
- The Rust binding requires `libsecdat >= 0.2.1` through `pkg-config`.

## Known Constraints

- `lock --save` persists supported volatile overlay changes, but persisted tombstone removal and v2 local-entry deletions still require a normal writable session.
- `secret status UUID` treats the UUID as an address, not authority; cross-domain linked objects should be inspected from the object-owning domain/store context.
